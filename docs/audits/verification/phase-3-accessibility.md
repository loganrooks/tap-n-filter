# Phase 3 Accessibility Audit

Phase 3 gate criterion 5 has two parts: (a) a programmatic accessibility-tree
check the verification subagent can re-run, and (b) a manual VoiceOver pass.
This document captures both parts and the orchestrator's analysis.

## Part (a): Programmatic accessibility-tree check

### Artifact

`test-artifacts/phase-3-accessibility-tree.json` — generated by the
`tap-n-filter-a11y-dump` executable target (see ADR-011 for the
artifact-producer / CI-gate split). The dump captures the AppKit
accessibility shadow tree of `ControlPanelView` with one EQ and one Reverb
in the graph.

Run to regenerate:

```sh
swift run tap-n-filter-a11y-dump
```

The `Tests/AccessibilityTreeTests` SwiftPM test target validates the
committed artifact's structural counts and scans `Sources/UI/*.swift` for
empty `.accessibilityLabel("")` / `.accessibilityValue("")` literals.

The low `nodesWithLabel` count is expected per ADR-011: SwiftUI's
`.accessibilityLabel(_:)` modifier does not consistently populate
`AXDescription` on the in-process KVC tree. Only `Add Effect` and
`Presets` (both `AXMenuButton`) surface labels via the KVC API. The
remaining interactive controls expose role + value correctly; their
labels are present in the SwiftUI source (verified in Part (a.2)) and
read aloud by VoiceOver (verified in Part (b)).

### Tab navigation order

`ControlPanelView` is a single `VStack(alignment: .leading, spacing: 0)`
arranged top-to-bottom; SwiftUI's default Tab order follows source order.

The expected Tab order, top to bottom:

1. `HeaderView` — status text (not focusable).
2. `SourcePickerView` — `AXPopUpButton` (audio source).
3. `ChainEditorView`
   - For each effect row (in order):
     - Expand/collapse chevron (`AXButton`)
     - Bypass toggle (`AXButton` / state-conveying)
     - Wet/Dry slider (`AXSlider`) — present on Reverb row, hidden on EQ row per ADR-007
     - Parameter sliders inside `EffectControlsView` when expanded (`AXSlider` × N)
     - Remove button (`AXButton`)
   - `Add Effect` menu button (`AXMenuButton`)
4. `FooterView`
   - `PresetMenu` (`AXMenuButton`)
   - `PowerToggle` (`AXButton` with state)

The order is deterministic and follows the visual top-to-bottom reading
order.

## Part (b): Manual VoiceOver pass

The orchestrator cannot drive VoiceOver on a freshly-built local development
binary via computer-use: the `request_access` allowlist is built from a
static installed-apps cache at session start, so a `.app` registered with
Launch Services mid-session is not granted (the compositor filters its
menubar icon out of screenshots, and clicks on it are blocked). The user
performs the VoiceOver pass directly.

### How to run the pass

A debug `.app` bundle has been installed at
`~/Applications/tap-n-filter.app` and registered with Launch Services.

1. **Launch the app:**

   ```sh
   open ~/Applications/tap-n-filter.app
   ```

   A menubar icon labeled "tap-n-filter" (or its symbol equivalent) should
   appear in the top-right of the screen.

2. **Enable VoiceOver:** press **Cmd-F5**. macOS announces "VoiceOver on."

3. **Open the control panel:** Move focus to the menu bar
   (**Cmd-F8**) or click the tap-n-filter menubar icon. VoiceOver
   announces the menu-bar item; press Return/Space to open the panel.

4. **Navigate the panel:** Use **Tab** (forward), **Shift-Tab**
   (backward), arrow keys (slider/picker adjustment), and Space (toggle
   buttons). For each control, note what VoiceOver announces — both the
   label and the current value (sliders/pickers should announce both).

5. **Disable VoiceOver:** press **Cmd-F5** again. macOS announces
   "VoiceOver off."

6. **Quit the app:** `pkill -f 'tap-n-filter'` or terminate via Activity
   Monitor.

### Observations

The orchestrator pre-filled the section below with the controls VoiceOver
should reach (deduced from the source-level accessibility modifiers).
The expected announcements describe what VoiceOver should read aloud
based on the source-level `.accessibilityLabel(_:)` /
`.accessibilityValue(_:)` modifiers.

| Control | Source modifier | Expected announcement |
|---|---|---|
| Source picker | `.accessibilityLabel("Audio source")` + `.accessibilityValue(displayName ?? "None")` | "Audio source, pop up button, None" (initial) |
| Add Effect button | `.accessibilityLabel("Add Effect")` | "Add Effect, menu button" |
| Effect-row expand chevron (EQ) | `.accessibilityLabel("Expand EQ" / "Collapse EQ")` | "Expand EQ, button" |
| Effect-row reorder up/down (EQ) | `.accessibilityLabel("Move EQ up"/"Move EQ down")` | "Move EQ up, button, dimmed" (Up disabled on first row) |
| Effect-row bypass toggle (EQ) | `.accessibilityLabel("EQ enabled")` + `.accessibilityValue("Active"/"Bypassed")` | "EQ enabled, button, Active" |
| Effect-row remove button (EQ) | `.accessibilityLabel("Remove EQ")` | "Remove EQ, button" |
| Effect-row expand chevron (Reverb) | `.accessibilityLabel("Expand Reverb" / "Collapse Reverb")` | "Expand Reverb, button" |
| Effect-row bypass toggle (Reverb) | `.accessibilityLabel("Reverb enabled")` + value | "Reverb enabled, button, Active" |
| Effect-row wet/dry slider (Reverb) | `.accessibilityLabel("Reverb wet/dry")` + `"N percent"` | "Reverb wet/dry, slider, 100 percent" |
| Effect-row remove button (Reverb) | `.accessibilityLabel("Remove Reverb")` | "Remove Reverb, button" |

### Findings

User ran the pass against the debug `.app` at
`~/Applications/tap-n-filter.app` on macOS 26.3 with VoiceOver enabled
(Cmd-F5) and a Bluetooth headphone output device.

- **VoiceOver navigation works.** Every interactive control was
  reachable via Tab / VoiceOver gestures and announced its label.
- A functional bug (unrelated to accessibility) was discovered during
  the same session: the live audio path did not produce the expected
  filtered output, because `AppViewModel.powerOn` routed the graph to
  `engine.outputNode` instead of `engine.mainMixerNode` and the tap
  was created without `muteBehavior = .muted`. Both fixes landed in
  the same commit run; ADR-014 documents the muting decision. The
  accessibility verdict is independent of that audio bug.

### Result

- **Result:** PASS (accessibility surface).
- **Date:** 2026-05-21.
- **macOS:** 26.3 (Build 25D125).
